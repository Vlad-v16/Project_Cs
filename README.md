# ♟️ Chess Game Online (WPF + .NET WebSockets)

**Chess Game Online** — це високопродуктивний клієнт-серверний застосунок для гри в шахи в режимі реального часу через мережу Інтернет. Проєкт реалізовано на базі сучасної платформи **.NET 9**. 

В основі архітектури лежить розділення на легковагий асинхронний сервер (Console Application) та інтерактивний графічний клієнт із багатим інтерфейсом (WPF — Windows Presentation Foundation). Зв'язок між вузлами системи забезпечується за допомогою низькорівневого двонаправленого протоколу **WebSockets**, що гарантує мінімальну затримку (latency) під час ігрового процесу.

---

## 🚀 Ключові архітектурні та технічні особливості

* **Низькоімпактний асинхронний ігровий цикл (Real-time):** Обмін даними здійснюється через постійне відкрите TCP-з'єднання. Сервер миттєво транслює (Broadcast) ходи гравців усім учасникам сесії, ліквідуючи потребу в неефективних HTTP-запитах опитування (Polling).
* **Високопродуктивний динамічний парсинг JSON:** Задля оптимізації роботи з пам'яттю та відмови від надлишкових класів-контейнерів (DTO), сервер аналізує пакети динамічно за допомогою класу `JsonDocument`. Рядки нотації ходів (наприклад, `e2 -> e4`) парсяться на льоту, знижуючи навантаження на збирач сміття (Garbage Collector).
* **Багатопотокова обробка сесій (`Task.Run`):** Клієнтські підключення ізольовані одне від одного. Кожен новий веб-сокет передається в окрему асинхронну задачу в пулі потоків (Thread Pool). Це гарантує, що затримки або блокування в одного користувача ніяк не вплинуть на стабільність інших гравців.
* **Потокобезпечне адміністрування станів (`ConcurrentDictionary`):** Пул підключених клієнтів та інформація про них (`PlayerInfo`) зберігаються в колекції `ConcurrentDictionary`. Це унеможливлює виникнення критичних помилок узгодження даних (Race Conditions) при одночасних запитах (наприклад, якщо один гравець робить хід, а інший у цей же момент здається чи відключається).
* **Динамічний статус-менеджер (Ролі та Глядачі):** Сервер автоматично керує фазами підключення:
  1. Перший підключений клієнт отримує роль `White Player` (Білі).
  2. Другий підключений клієнт отримує роль `Black Player` (Чорні).
  3. Усі наступні підключення автоматично маркуються як `Spectator` (Глядачі) з обмеженням прав (можливість лише переглядати гру в реальному часі без права відправки ходів).
* **Стійкість до розривів зв'язку (Fault Tolerance):** У системі реалізовано централізований обробник подій життєвого циклу гри. Сервер чітко диференціює навмисний вихід гравця (кнопка *Resign*) від аварійного обриву сокета або закриття застосунку (*Disconnect*), автоматично присуджуючи технічну перемогу опоненту та сповіщаючи всіх учасників.

## 📂 Структура та компоненти проєкту

Проєкт спроєктовано за модульним принципом і розділено на два ключові блоки:

```text
├── ChessServer/               # Серверний модуль (Консольний застосунок)
│   ├── Models/                # Об'єктно-орієнтована модель шахів
│   │   ├── Board.cs           # Логіка шахівниці, генерація валідних ходів
│   │   ├── Pieces/            # Фігури (Rook, Knight, Bishop, Queen, King, Pawn)
│   │   └── Move.cs            # (Опціонально/Архів) Контракти структури ходу
│   └── Program.cs             # Точка входу сервера: HttpListener, WebSocket-сервер, Broadcast-логіка
│
└── ChessGameUI/               # Клієнтський модуль (Графічний WPF-застосунок)
    ├── App.xaml               # Глобальні ресурси, стилі кнопок та конфігурація StartupUri
    ├── App.xaml.cs            # Логіка управління життєвим циклом клієнтського додатка
    ├── AssemblyInfo.cs        # Керування параметрами локалізації та пошуку ресурсів тем WPF
    ├── MainWindow.xaml        # Візуальна розмітка інтерфейсу: сітка дошки, TextBox імені, кнопки керування
    └── MainWindow.xaml.cs     # Code-behind: асинхронна робота з ClientWebSocket, рендеринг фігур, події кліків
```

# 🛠️ Технологічний стек

* Платформа: .NET 9.0 (інтерфейс Windows-специфічний через використання WPF)
* Мова програмування: C# 13
* Графічний стек: WPF (XAML, механізми подій, динамічний рендеринг елементів UI)
* Мережеві протоколи: Веб-сокети (System.Net.WebSockets, HttpListenerWebSocketContext)
* Серіалізація даних: System.Text.Json (використання JsonSerializer та JsonDocument)
* Інструмент прокидання портів: ngrok (тунелювання TCP/HTTP трафіку для обходу NAT)

# 💻 Розгортання та запуск у локальній мережі (Localhost)
Для локального тестування (на одному комп'ютері або всередині однієї Wi-Fi мережі) виконайте наступні кроки:

Крок 1: Компіляція та запуск сервера
Відкрийте термінал у корені проєкту, перейдіть до папки сервера та запустіть його:

```json
cd ChessServer
dotnet run
```

* Після успішного старту в консолі з'явиться повідомлення: Сервер шахів успішно запущено на порту 8080...

Крок 2: Запуск клієнтських ігрових вікон
Для повноцінної гри необхідно запустити мінімум два екземпляри клієнта. Відкрийте два окремих вікна терміналу й виконайте:

```json
cd ChessGameUI
dotnet run
```

* Налаштування підключення: В обох вікнах у полі Server залиште адресу за замовчуванням: ws://localhost:8080/.
* Введіть унікальні імена у полі Your name та натисніть кнопку CONNECT.

# 🌐 Налаштування гри через Інтернет (За допомогою ngrok)
Якщо гравці знаходяться за різними роутерами (в різних мережах/містах), для підключення без використання білої IP-адреси використовується утиліта ngrok.

1. Прокидання тунелю з підміною заголовка
Оскільки локальний .NET HttpListener суворо валідує заголовок Host і очікує запити на localhost, запуск звичайного HTTP-тунелю призведе до помилки 400 Bad Request. Щоб обійти це обмеження, запустіть ngrok із прапорцем підміни Host-заголовка:

```json
ngrok http 8080 --host-header=localhost:8080
```

2. Отримання та адаптація адреси
У вікні ngrok знайдіть рядок Forwarding. Він буде виглядати приблизно так:

```json
https://1234-56-78.ngrok-free.app
```

3. Конфігурація на стороні клієнта
Гравці копіюють це посилання, вставляють його в полі Server у додатку WPF, але обов'язково змінюють префікс протоколу з https:// на безпечний протокол веб-сокетів — wss://:

Приклад для введення в інтерфейсі:

```json
wss://1234-56-78.ngrok-free.app
```

# 📝 Специфікація мережевого JSON-протоколу (API)
Обмін даними між клієнтом і сервером стандартизовано у форматі текстових JSON-пакетів, що містять обов'язкове кореневе поле "type" та об'єкт контенту "data".

1. Авторизація клієнта (player_connect)
* Клієнт -> Сервер: Надсилається відразу після натискання кнопки CONNECT.

```json
{
  "type": "player_connect",
  "data": { "playerName": "Vlad" }
}
```

* Сервер -> Клієнт (connect_success): Підтвердження авторизації та призначення кольору фігур.

```json
{
  "type": "connect_success",
  "data": {
    "playerId": "4b6a221f-829d-4c31-b3b4-e3500f4021aa",
    "color": "white"
  }
}
```

2. Запит доступних ходів (get_valid_moves)
* Клієнт -> Сервер: Викликається при кліку на фігуру для отримання підсвічування клітинок.

```json
{
  "type": "get_valid_moves",
  "data": { "position": "e2" }
}
```

* Сервер -> Клієнт (valid_moves): Повертає масив доступних координат із прапорцем, чи є хід взяттям чужої фігури.

```json
{
  "type": "valid_moves",
  "data": {
    "moves": [
      { "to": "e3", "isCapture": false },
      { "to": "e4", "isCapture": false }
    ]
  }
}
```

3. Здійснення ходу фігурою (chess_move)
* Клієнт -> Сервер: Відправляється, коли гравець перетягує або клікає фігурою на цільову клітинку.

```json
{
  "type": "chess_move",
  "data": { "from": "e2", "to": "e4" }
}
```

* Сервер -> Клієнти (Broadcast move_made): Транслюється обом гравцям та глядачам для синхронізації дошки. Включає перевірку на шах/мат.

```json
{
  "type": "move_made",
  "data": {
    "from": "e2",
    "to": "e4",
    "check": "none",
    "status": "active"
  }
}
```

4. Подія здачі або дисконекту (player_resigned)
* Сервер -> Клієнти (Broadcast): Розсилається, якщо гравець натиснув кнопку RESIGN або закрив вікно програми.

```json
{
  "type": "player_resigned",
  "data": {
    "leaverId": "black",
    "leaverName": "Black Player",
    "winnerName": "Vlad",
    "reason": "resign" 
  }
}
```

* (Значення поля reason може бути "resign" або "disconnect").
